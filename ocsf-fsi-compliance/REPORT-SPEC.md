# BENCH-F — §1003(b) report spec (frozen)

**Frozen before any timing run.** This fixes the report sections, their acceptance
definitions, the timing protocol, and the decision rule, so the human-hours result can't
be retuned to the ≥40% reduction the hypothesis would like. The corpus and its planted
intrusion are generated by `gen_corpus.py` (seeded, reproducible) and are identical for
both substrates.

## What this measures, and the honesty problem up front

H-FSI-01: SEC Reg SCI §1003(b) annual review is structurally a forced data-engineering
exercise — the entity has to demonstrate operational resilience *with evidence* (logs,
lineage, time-travel queries, reproducible BC/DR artifacts). The claim is that producing
that evidence costs fewer human-hours on an Iceberg + catalog + time-travel lakehouse than
on a schema-on-read SIEM + document store + manual export. The headline metric is
**human-hours per review cycle**, which is a labor measurement, not a machine metric — and
it is confounded, because the same operator who would build the lakehouse also times the
report and has a commercial incentive for the lakehouse to win. The protocol below spends
its rigor budget on that confound, and the result is labelled an upper-bound-of-favorability
for the lakehouse, not a neutral estimate. A synthetic, self-timed human-hours number is the
weakest evidence in the lab; that is stated in the headline, not a footnote.

## The two substrates

- **SIEM substrate** — a schema-on-read SIEM for search, a document store for the narrative,
  and manual export/screenshot for evidence artifacts. Built to genuine competence (native
  reporting features, not a strawman manual workflow).
- **Lakehouse substrate** — Iceberg + a catalog (e.g. Polaris) + Trino/DuckDB, with snapshot
  time-travel and catalog lineage enabled.

Both are built to a comparable level of finish *before* any timing run.

## The report sections (frozen acceptance definitions)

Each section is one concrete evidence-production task with an unambiguous done-definition,
so the human-hours measure the same deliverable on both substrates.

- **(a) Retention attestation.** Demonstrate that security telemetry for a named system has
  been retained for the §1005 five-year / 17a-4 six-year floor, with the first two years
  readily accessible. *Accept:* a produced artifact showing per-year partition coverage for
  the named system back to the retention floor, plus a readily-accessible check on the most
  recent two years. Ground truth: `retention` in `ground_truth.json`.
- **(b) Incident-timeline reconstruction.** Produce the ordered who-did-what-when timeline for
  the planted intrusion (the §1002 notification-window evidence shape). *Accept:* the ordered
  event sequence matching `intrusion_timeline` by true event time. Ground truth:
  `intrusion_timeline`.
- **(c) Lineage / chain-of-custody.** For a sample of records, show where each came from and
  whether it has been altered. *Accept:* for each `lineage_sample_uids` record, a
  provenance/integrity artifact. Ground truth: `lineage_sample_uids`.
- **(d) BC/DR point-in-time reproduction.** Re-run a query as of a past instant (time-travel).
  *Accept:* the query result as of `point_in_time_ms`, reproducible. Ground truth:
  `point_in_time_ms`.

## Timing protocol (the rigor budget)

For each section × substrate, record operator wall-clock time-on-task with a stopwatch:
start when the section's inputs are available, stop when its frozen acceptance definition is
met. To control the build-it-then-time-it confound:

- pre-register the section list and acceptance definitions here, before either substrate is built;
- build **both** substrates to comparable finish before any timing run;
- run the sections in **counterbalanced order** across substrates, so the learning effect
  (getting faster at the *task* on the second substrate) is distributed, not always favoring
  whichever goes second;
- separate the pipeline-author from the report-operator where feasible, or document that they
  are the same person and treat the number as a self-timed upper-bound-of-favorability;
- record **hours by section**, because the hypothesis predicts the gap concentrates in the
  lineage/time-travel sections (c)+(d) — a *uniform* gap across all sections is a red flag for
  an operator-bias artifact, not a substrate effect.

The one-time lakehouse build cost is excluded from the per-cycle human-hours (the claim is
about recurring annual-review cost), and that exclusion is stated because it favors the
lakehouse — the SIEM incumbent has already amortized its build.

## Defensibility, scored separately

"Cheaper" and "more defensible" are two different claims and must not be blended. Per section,
record a binary defensibility verdict: does the produced evidence carry machine-checkable
provenance (Iceberg snapshot time-travel + catalog lineage), or is it a manual export /
screenshot? `defensibility.py` auto-assesses the lakehouse side (the substrate either provides
time-travel/lineage for the section or it doesn't); the SIEM side is the operator's to record.

## Decision rule (pre-committed)

- **Confirm** (supports H-FSI-01): the lakehouse produces the full report in ≤60% of the SIEM
  human-hours (the ≥40% reduction the hypothesis names), AND the reduction concentrates in the
  lineage/time-travel sections (c)+(d) rather than being uniform.
- **Disconfirm / null** (the hypothesis's own falsification line): the reduction is <20% — the
  SIEM evidence-production is already ergonomic for this report shape.
- **Inconclusive**: 20–40%, or a reduction that is uniform across sections (an operator-bias
  artifact rather than a substrate effect).

## What stays out of scope

Not a legal opinion that either substrate *satisfies* Reg SCI or 17a-4 (no first-mover
precedent exists — H-FSI-02). Not a migration-cost or migration-risk benchmark; the §1002
evidence-integrity obligation can make an entity reluctant to disturb a working audited trail
mid-cutover, so cheaper evidence-production does not mean cheaper or safer migration, and the
result must not be read as "therefore migrate." Not a per-TB retention-cost benchmark (H-FSI-03,
a sibling).

## Evidence tier

Tier B at best on the synthetic run, and arguably weaker: synthetic corpus, single operator,
self-timed human-hours with the build-then-time confound only partially controlled. The
regulatory text inputs (§1003(b), §1005, 17a-4(b)) are Tier A as *requirements*; the
human-hours *result* is Tier B/C. Tier-A promotion needs a named-FSI run with the entity's own
operators and corpus — explicitly gated, and a paid engagement, kept distinct from this public
synthetic spec.
