# BENCH-E findings — DuckLake catalog failure modes (2026-06-14)

First scored run of BENCH-E, the catalog-failure-mode bench that came out of the
Iceberg-V4-vs-DuckLake metadata work. Three verified-real, OPEN `duckdb/ducklake` issues,
reproduced (or not) on the current stack with versions recorded per arm. Per-arm JSONs in
`results/`; `results/summary.json` is the machine-readable headline; `README.md` has the full
method.

## What this is for

The DuckLake pitch leads with planning speed, and the V4/DuckLake reconciliation already
established that its headline 105× streaming number is an unequal-workload comparison I should not
reproduce as-is. The fair-broker move is not to argue the marketing — it is to measure what a
security team actually hits when it stands DuckLake up on a Postgres catalog. These three issues
are that: two open correctness/operability gaps, and one that a point release quietly fixed. None
of this is vendor-bashing; it is the same treatment the lab gives every engine, and DuckLake's
SQL-catalog design is genuinely interesting — which is exactly why the catalog-layer bugs are
worth pinning to versions.

## The findings

**E1 (#1215) — cross-store delete-conflict gap — PERSISTS on DuckDB 1.5.3 + DuckLake e6a3bd0a.**
Two concurrent deletes of the same row commit without conflict when one is inlined (≤
`DATA_INLINING_ROW_LIMIT`) and the other is a parquet delete file (> the limit); the conflict
check never compares the two stores. The deleted rows then reappear: in both commit orders the
repro leaves **29 rows** that a correct system would have deleted to **0**, with no error raised.
This is a silent correctness bug — a lost write-conflict that resurrects deleted data — and it is
live on the current extension. For a security lakehouse this is the worst class of bug: a deletion
(think GDPR erasure, or retention expiry, or a tombstoned false-positive) can silently not take
effect under ordinary concurrent SQL.

**E2 (#1184) — >1600-column CREATE TABLE wall on a Postgres catalog — PERSISTS.** DuckLake
unconditionally creates a backing `ducklake_inlined_data_<id>_0` table carrying every user column
as `BYTEA`, which hits Postgres's hard 1600-column-per-table limit, so a wide table cannot be
created at all — and `SET ducklake_default_data_inlining_row_limit = 0` does not help, because the
backing schema is still emitted with all columns. Measured cleanly: 1500 columns creates fine;
1600 and 1700 both fail with `ERROR: tables can have at most 1600 columns`; the inlining-disabled
1700-column case fails identically. Security schemas go wide — a fully-flattened OCSF event with
all profiles and observables, or a normalized EDR/firewall dataset, can exceed 1600 columns — so
this is a real ceiling on the schema-on-write path when the catalog is Postgres.

**E3 (#1031) — Postgres connection-pool timeout — FIXED between 1.5.2 and 1.5.3.** This is the
version-currency headline. On **DuckDB 1.5.2** (where the DuckLake extension auto-resolves to
**415a9ebd**, the exact commit named in the issue), creating 60 tables then one
`information_schema.tables` query hangs **60 s** and fails with `Connection pool timeout: all 14
connections in use, waited 30000ms` — and the pool exhausts at **14** connections, one per worker
thread on this 14-thread host, which is precisely the thread-local-caching mechanism the issue's
root-cause analysis described. On **DuckDB 1.5.3** (DuckLake e6a3bd0a) the identical workload
returns in **0.036 s** with no timeout. The issue is still labeled OPEN with "PR submitted"
upstream; the controlled comparison shows the fix is effectively in the 1.5.3 release. The lesson
is the method, not the bug: "probably fixed" is worthless without running both versions, and the
old-version confirmation is what separates a real fix from a repro that simply under-triggers on
this host.

## So-what for the literature and hypotheses

- **H-DUCKLAKE-02 / the Iceberg-V4-vs-DuckLake essay:** the honest framing is now measured, not
  asserted. DuckLake's SQL catalog buys planning speed and pays for it with catalog-layer
  correctness/operability surface — two gaps open today (#1215 silent delete resurrection, #1184
  wide-schema wall) and one regression already closed by a point release (#1031). The essay should
  cite the verified issue numbers, the per-version verdicts, and the discipline (don't reproduce
  the 105× claim; do reproduce the failure modes). Keep it peer-level and version-bound.
- **The wide-schema wall (#1184) is the most security-relevant** because OCSF-flattened events and
  normalized security tables routinely exceed 1600 columns, and the schema-on-write path through a
  Postgres-backed DuckLake catalog cannot represent them at all. That is a concrete architectural
  constraint for anyone weighing DuckLake as a security-lakehouse catalog, and it pairs with the
  flattening-fidelity / nested-OCSF benches that already document the cost of going wide.
- **The version-currency method generalizes** (and matches B-ANSWEREQ chDB): every "known bug"
  claim in the literature should be re-pinned to a current version before it is repeated, and a
  "fixed" claim needs the old version reproducing as its control. This is now demonstrated twice.

## Scope, honestly

Single host, the issues' own minimal repros, versions recorded per arm. E1 is a boolean
correctness result (no timing); E2 is a clean boundary probe; E3 is a controlled 1.5.2→1.5.3
delta. Catalog-layer only — these say nothing about DuckLake read/scan performance. The verdicts
are version-bound: re-run on the next DuckLake release before repeating them, especially #1215 and
#1184, which are open and may close the same way #1031 did.
