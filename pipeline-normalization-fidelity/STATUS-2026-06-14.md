## The Tenzir coverage map (this IS P1 — shipped-mapping availability)

Tenzir's Community Library (github.com/tenzir/library) publishes OCSF mappings as TQL
packages. Against this bench's four sources:

| source | Tenzir shipped mapping | faithful on THIS corpus? |
|---|---|---|
| zeek_conn | `zeek::ocsf::map` → conn → Network Activity | **yes** — reads Zeek JSON; corpus field names match (id.orig_h, proto, conn_state, uid, ts) |
| sysmon | `sysmon::ocsf::map` | **no** — the operator calls `parse_winlog()`, i.e. it expects raw Windows EVTX/XML, not the pre-parsed JSON the corpus carries (CommandLine/Image/ProcessGuid). A coverage gap for JSON-shaped Sysmon |
| cloudtrail | none | **no** — the `amazon` package ships route53 / security_lake / vpc_flow only; there is no CloudTrail-management-events → OCSF API Activity operator |
| auth | none | **no** — generic auth source by design; no vendor ships a mapping for these field names |

So Tenzir faithfully scores **Zeek conn** on this corpus; sysmon/cloudtrail/auth are
recorded as 0% coverage per the README stop rule — and that availability gap is exactly
prediction P1. (To score sysmon faithfully you would synthesise raw winlog XML from the
fields, which transforms the input rather than measuring the shipped JSON path; left out
as not faithful.)
