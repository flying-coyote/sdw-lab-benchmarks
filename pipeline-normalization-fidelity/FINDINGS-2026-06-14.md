# Pipeline normalization-fidelity (#10) — Tenzir arm scored (2026-06-14)

First scored arm of the bench. Tenzir's **shipped** library OCSF mapping was run, unedited,
over the pinned corpus; this is what the tool's published mapping actually delivers, not a
hand-written one. Tier B, single host. Versions pinned: Tenzir **6.0.0**, library commit
**671e049** (`zeek::ocsf::map`), OCSF **1.8.0**. Per-source scores in `results/results.json`
+ `results/RESULTS.md`; method + the blocked-then-unblocked history in `STATUS-2026-06-14.md`.

## Result — Tenzir shipped mappings vs the gold

| source | OCSF class | coverage | field | value | class | activity_id |
|---|---|--:|--:|--:|--:|--:|
| zeek_conn | network_activity 4001 | 100% | 80% | 92% | **100%** | **17%** |
| cloudtrail | api_activity 6003 | 0% | — | — | — | — |
| sysmon | process_activity 1007 | 0% | — | — | — | — |
| auth | authentication 3002 | 0% | — | — | — | — |

**Zeek conn (the one source Tenzir ships a JSON-consumable mapping for):** the mapping nails
the OCSF *class* (100% — correctly Network Activity 4001) and most *values* (92%), covers every
gold field somewhere (100% coverage), but two real gaps show up that field-level scoring alone
would hide:
- **activity_id 17%** — the mapping does not derive the OCSF activity from Zeek's `conn_state`,
  so on 83% of records the activity classification diverges from the faithful gold. A consumer
  filtering on `activity_id` (open vs close vs reset vs fail) would mis-bucket most connections.
- **field fidelity 80%** — `history`, `service`, and `uid` land in `unmapped` rather than a typed
  OCSF attribute (each on all 100k records). `value` 92% is mostly the `ts` representation (the
  mapping's OCSF time differs from the gold's canonical value on every record) plus a few
  `duration` cases.

**The other three sources are 0% coverage — and that is the P1 finding, not a harness gap.**
Tenzir's Community Library ships OCSF mappings for Zeek and Sysmon and an `amazon` package, but:
- **cloudtrail** — the `amazon` package ships route53 / security_lake / vpc_flow only; there is
  no CloudTrail-management-events → OCSF API Activity operator, so nothing maps.
- **sysmon** — the shipped `sysmon::ocsf::map` calls `parse_winlog()`; it expects raw Windows
  EVTX/XML, not the pre-parsed JSON the corpus (and most EDR shippers) carry, so it does not
  consume this input shape.
- **auth** — a generic auth source by design; no vendor ships a mapping for these field names.

**Caveat on the failure-class table** (`results/RESULTS.md`): the "100%" failure-class cells on
the three 0%-coverage sources are no-output artifacts (with nothing emitted, every probe
trivially "reproduces"), not real reproductions — only the zeek_conn row (entity_role 0%,
observables_flattening 0%) is a genuine measurement, and there Tenzir does NOT reproduce those
crosswalk failure classes.

## So-what

- **For Matrix Component 4 ("OCSF normalization at-source"):** Tenzir's shipped mapping is real
  and gets the easy 80% (class + most values) but leaves a measurable semantic gap on the part
  that matters for detection — the activity classification — and ships no mapping at all for two
  of four common security sources in their consumable shape. "Maps to OCSF" is a coverage claim,
  not a fidelity guarantee, and this bench is the measurement that separates the two.
- **P1 (coverage ordering / availability) is the headline:** shipped-mapping availability is the
  binding constraint, not per-field fidelity — three of four sources have no usable shipped
  Tenzir mapping. Cribl and Vector arms (deferred — Cribl needs an account, Vector is mostly
  non-official VRL) would test whether that availability gap is tool-specific or general.

## Scope, honestly

One tool (Tenzir), one faithful source (Zeek conn), shipped mapping run unedited, versions
pinned, single host. The percentages ride with the synthetic corpus (realistic field names,
seeded, not your estate); what travels is the shape — class-right/activity-wrong on Zeek, and
the shipped-mapping coverage gaps. The activity_id and field gaps are version-bound to library
commit 671e049; re-run on a newer library before repeating. Cribl + Vector arms remain to run.
